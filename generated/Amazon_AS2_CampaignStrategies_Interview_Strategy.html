⑫ 知识速查 A — DPO & PPO 从头回忆（你做过但忘了）

先记住一张图：RLHF 的三个阶段，PPO 在第三段，DPO 把三段压成一段。

① SFT（监督微调）
  • 用人工标注的"好回答"示范数据
  • 让模型学会"像人一样回答"
  • 结果：会说话，但不一定说对话
数据：(prompt, good_response) pairs
② Reward Model 训练
  • 人类比较两个回答，选好的那个
  • 训练一个打分模型 R(x, y)
  • 结果：知道什么叫"好"
数据：(prompt, y_win, y_lose) pairs
③ PPO（强化学习优化）
  • 用 RM 分数作为 reward
  • PPO 更新 policy，让分数更高
  • KL 惩罚防止跑偏太远
同时跑：actor, critic, RM, reference 四个模型
PPO 核心机制

为什么叫 Proximal？ — 每次更新不能走太远。Loss 里有个 clip：

L = min( r_t · A_t,   clip(r_t, 1-ε, 1+ε) · A_t )
r_t = π_θ(a|s) / π_old(a|s)  — ratio 新旧 policy
A_t = advantage = R - V(s)  — 比预期好多少
实际训练时的问题：
  • 要同时维护 4 个模型（actor, critic, reward, reference）→ 显存爆炸
  • Reward hacking：模型学会"骗"reward model，说流畅废话
  • 训练不稳定，KL 系数敏感
  • 需要大量 rollout 样本 → 慢

面试一句话： "PPO 是 on-policy RL，训练稳定但代价高，4 个模型同时驻留 GPU，数据效率低。"

DPO 核心机制（最重要，背这个）

DPO 的关键 insight： 最优 policy 和 reward model 之间有个解析关系，可以把 reward model 消掉，直接从 preference 数据优化 policy。

L_DPO = -log σ ( β · log π_θ(y_w|x)/π_ref(y_w|x)  −  β · log π_θ(y_l|x)/π_ref(y_l|x) )

y_w = chosen (好的回答)  |  y_l = rejected (差的回答)
第一项：让好回答相对 reference 概率↑
第二项：让差回答相对 reference 概率↓
β 的含义（面试必问）：
  • β 大 → 越紧贴 reference model，更保守（不容易出奇怪输出）
  • β 小 → 偏离 reference 越远，更灵活但风险大
  • 典型值：β = 0.1 ~ 0.5
DPO vs PPO 优缺点对比：
  • ✅ DPO：只需 2 个模型（policy + reference），训练稳定，数据效率高
  • ✅ DPO：离线训练，不需要 rollout
  • ❌ DPO：对 preference 数据质量极度敏感（噪声 pair 会严重伤害性能）
  • ❌ DPO：无法做 online exploration，学不到数据之外的行为
  • ❌ PPO：计算贵，但理论上能做 online RL，探索能力更强

面试一句话： "DPO 把三步变一步，用对比 loss 直接优化 policy，简单稳定，但 preference 数据质量是瓶颈。"

DPO 变体速查（认识就行，不用背细节）
IPO
修复 DPO 的理论缺陷（DPO 在 preference 完全分离时退化）。Loss 加了 正则项。Amazon JD 点名了 IPO。
SimPO
不需要 reference model，用序列平均 log prob 代替 ratio。更简单，省显存。
ORPO
把 SFT loss 和 preference loss 合并成一步，连 reference model 也不需要。
GRPO（DeepSeek 用的）
Group Relative Policy Optimization，把同一 prompt 的多个输出做组内对比，不需要独立 value function。

面试中提到 IPO（因为 JD 点名了）+ GRPO（展示你跟进最新进展）就够了。

⑬ 知识速查 B — Tool Calling 深度 & 你的核心研究立场
Tool Calling 工作原理（从底层理解）

Tool calling 不是魔法，是一个精确的 structured output → execution → injection 循环：

1. 开发者定义工具 schema（名字、参数、描述）
2. Schema 注入 system prompt 或特殊 token 序列
3. LLM 生成结构化 JSON：{ "tool": "get_campaign_stats", "args": {"id": 123} }
4. 框架捕获 JSON，验证 schema，执行工具
5. 工具结果作为 tool message 注回 context
6. LLM 继续生成，based on 工具结果推理下一步
关键设计决策：
  • 并行 vs 串行调用：OpenAI / Anthropic 支持并行 tool call（同一轮生成多个 JSON），适合独立任务；串行适合有依赖的步骤
  • Tool schema 质量：描述越精确，模型越不容易选错工具。这是被大量低估的工程工作。
  • Context 管理：每次 tool result 都塞进 context，多轮后 context 快速膨胀，需要压缩/摘要策略
  • 安全边界：工具执行 side effects（改数据库、暂停活动）需要 confirmation gate 或 dry-run 模式
MCP（Model Context Protocol）架构

MCP 是 Anthropic 2024 年发布的开放标准，把 tool 的定义、发现、调用标准化成一个协议。

MCP Server（工具侧）
  • 暴露 Tools（可调用函数）
  • 暴露 Resources（可读数据）
  • 暴露 Prompts（模板）
  • 通过 list_tools() 让 client 发现
MCP Client（Agent 侧）
  • 连接 server，拉取工具列表
  • 把工具注入 LLM context
  • 把 LLM 的 tool call 路由到正确 server
  • 拿结果返回给 LLM
"MCP 的核心价值不是让 tool call 更容易，而是让工具生态标准化——任何 MCP-compatible 工具都可以插入任何 MCP-compatible agent，不需要重新训练或重新提示。这对 Amazon 的广告工具生态（auction API, keyword API, analytics API）非常有价值。"
🔥 Tool Call Failure 分类学（你的核心研究兴趣，面试加分）

这是目前 agent reliability 领域最活跃的研究方向。把 failure 分成两大类，七种模式：

一、模型侧 Failure（模型的锅）
① Wrong Tool Selection
任务需要 A，模型选了 B。通常因为 tool description 写得不够清晰，或模型没理解意图。
📍 广告场景：用 "get_keywords" 代替 "get_search_terms"
② Bad Parameter Generation
工具选对了，但参数错了——类型不符、缺少必填字段、超出范围、注入式字符串。
📍 广告场景：bid 参数传了负值或字符串
③ Hallucinated Tool Call
调用了不存在的工具，或使用了不存在的参数字段。
📍 比例随 tool 数量增加显著上升
④ Loop / Over-calling
工具失败后不断重试，或在不需要工具的时候也调用。缺乏 "don't call if uncertain" 机制。
📍 生产环境：引发 API 限速或账单问题
二、系统侧 Failure（环境的锅）
⑤ Execution Failure
工具本身报错：网络超时、rate limit、下游服务不可用、权限不足。
📍 模型必须能区分"我的调用错了"vs"系统坏了"
⑥ Stale / Inconsistent Results
工具返回了过期数据或中间状态数据，模型基于错误信息做决策。
📍 广告场景：竞拍价格数据 lag 导致错误出价建议
⑦ State Drift in Multi-step
多步执行中，step N 的 side effect 让 step N+2 的假设失效。Agent 对世界状态的 mental model 和实际状态不同步。
📍 最难处理的 failure，需要 state tracking
Failure 处理机制 — 研究前沿
✅ 现在工业界在用的（成熟）
  • Schema validation before execution — 在 LLM 调用工具之前，用 JSON Schema 验证参数合法性，非法就打回让模型重生成
  • Retry with error injection — 把错误信息注回 context，让模型看到具体报错信息再重试
  • Max retry + fallback — 超过 N 次失败后降级：用更简单工具 / 不用工具直接回答 / 转人工
  • Confirmation gate for irreversible actions — 改数据、删内容类的操作先 dry-run 或要求用户确认
🔬 研究前沿（面试加分）
  • Reflexion-style self-correction — agent 看到失败后生成"为什么失败"的 verbal reflection，指导下次重试（你做过！）
  • Tool retrieval / RAG for tools — 工具太多时，先检索最相关的 K 个工具再注入 context，减少选错概率（CRAFT 论文）
  • Uncertainty-aware calling — 在生成 tool call 前估计 confidence，低于阈值时改为 "ask for clarification" 而不是盲目调用
  • State snapshot + rollback — 记录每步执行前的系统状态，失败时 rollback（类似数据库事务）
  • Multi-agent supervision — 用监督 agent 专门检查 worker agent 的 tool call 是否合理，拦截异常

✅ 正确版本（有据可依的科学立场）
"I think we're at an interesting inflection point. Base model capabilities — reasoning, instruction following, code generation — have improved dramatically through pre-training and alignment techniques like DPO. We're seeing diminishing marginal returns from optimizing those algorithms further, at least at the model-behavior level.

What I find more tractable and higher-ROI right now is reliability engineering for agentic systems — specifically, how agents handle tool failures, maintain consistent world-state across multi-step execution, and recover gracefully when the environment behaves unexpectedly. For a system guiding 1.6 million advertisers, a tool failure that silently corrupts campaign state is a much more dangerous failure mode than a model that sounds slightly less helpful.

That said, DPO and preference optimization are absolutely still important — they're what make the model's tool-use decisions trustworthy in the first place. A model that hallucinates tool calls or ignores tool output isn't fixable with retry logic. So I see training and reliability engineering as complementary layers: training ensures the model reasons correctly, reliability engineering ensures the system behaves correctly even when the model is uncertain or the environment fails."

追问准备："那你觉得 tool reliability 里最值得研究的方向是什么？"
"I'm most interested in state consistency in multi-step agent workflows. The hard problem isn't a single tool failure — it's when step 3 succeeds but changes the environment in a way that makes step 5's assumptions invalid, and the agent doesn't detect the drift. This is the agentic equivalent of a transaction isolation problem in databases. Current approaches like Reflexion are good at single-step error correction, but there's no good framework yet for tracking and validating world-state across an entire execution plan."
